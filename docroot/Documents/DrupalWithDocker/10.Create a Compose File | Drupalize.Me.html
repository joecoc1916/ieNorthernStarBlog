Create a Compose File

Remove from queue     Share
Last updated November 11, 2019
Backend and Infrastructure

Docker Compose allows us to manage several related containers as a single group. We define container sets by creating a creating the Compose file, docker-compose.yml.

In this tutorial, we'll:

  • Create the basic structure of the Compose file
  • Define a container set using off-the-shelf containers
  • Describe where to place it in your project
  • See how directory names are significant in Compose

Goal

Understand the structure and purpose of the Compose file.

Prerequisites

  • What is Docker?
  • Run Sets of Containers

What does project mean here?

In this tutorial, the term project refers to the root directory that contains files related to your web site. Depending on your set-up, this might be the same the web-accessible webroot or docroot of your project (where index.php lives) or it might be a directory one level above the webroot and therefore not web-accessible. This can be desirable in order to place configuration files, build scripts, or backup database dumps in a location associated with the project, but not web-accessible for security's sake. But in this tutorial, we refer to project to whatever the root directory of your web project's files, either web-accessible or in a directory containing a web-accessible webroot or docroot.

Example 1: Project root outside of web-accessible docroot directory.

my-web-site-project
- .git
- .docker-compose.yml
- build-scripts/
- docroot/
  - index.php

Example 2: Project root same as web-accessible directory.

my-web-site-project
- .git
- .docker-compose.yml
- build-scripts/
- index.php

File name project placement

When creating a Docker Compose file, it is typically saved as docker-compose.yml in the root of your project's directory. While the Compose file can be named anything, it expects this name by default. Most projects only need one Compose file to support a local development environment, so this makes sense.

Furthermore, it is best to always place the docker-compose.yml file under version control. That way, any changes to the file can be tracked, and all developers can share the same configuration of containers for the project. This cuts down on it-works-on-my-machine problems by standardizing the project's development infrastructure. Since the Compose file is just YAML, it is trivial to add to a repository.

The project directory name is significant!

Always place the docker-compose.yml file in the project root because of the way Compose auto-generates container names. While the container IDs are always unique, containers also have a container name that is human-readable. This name is only unique to each system; two containers running on different machines will have different container IDs, but can have the same name.

Docker Compose relies on the name of the directory containing docker-compose.yml to name containers. The directory containing a project on your system usually also has a unique name. As a result, it is both best practice (and easier!) to place the Compose file in the root of your project directory.

Specifying the Compose file version

Almost all Compose files today start with the Compose file version:

version: '3'

The version key specifies the Compose file format version. It does not specify the Docker version, Docker Compose version, or your version of the file. At the time of this writing, the most recent version of the Compose file format is 3.

The earliest versions of the Compose file format did not include the version key. As a result, the version key must be at the beginning of the file, and there must be only one version key per file.

The services section

The bulk of a docker-compose.yml file is the services section. This section describes the containers and configurations to use for those containers. It starts with the services key, typically defined right below the version key:

version: '3'
services:

Under the services key, we define each container that's part of the set. Each container is identified by a human-readable name. As the Compose file is in YAML format, indentation is significant:

version: '3'
services:
  my_first_container:

The above, however, isn't a valid file. Docker Compose, just like docker run, needs to know the image name. We specify that with the image key:

version: '3'
services:
  my_first_container:
    image: debian

Additional services can be added by simply appending them to the end of docker-compose.yml. Again, remember that indentation is significant:

version: '3'
services:
  my_first_container:
    image: debian
  my_second_container:
    image: debian
  my_third_container:
    image: debian

A basic web stack

The above example compose file isn't very useful. After all, the debian image only has a base operating system in it. There's no web server applications on it like Apache, PHP, or MySQL. Fortunately, there are many, many container images we can use; many particularly useful to us as Drupal developers. Let's put together a basic web stack in Compose:

version: '3'
services:
  web:
    image: osiolabs/drupaldevwithdocker-php
  db:
    image: osiolabs/drupaldevwithdocker-mysql

The above has 2 containers in it:

  • The web container uses an image that was created specifically for this series, and provides Apache and PHP 7.
  • The db container also uses an image that was created specifically for this series. It provides a MySQL server.

Recap

In this tutorial we described how to create a container set in Docker Compose. The docker-compose.yml file lets us define one or more containers under the services section. Finally we described a basic web stack in Compose using off-the-shelf images.
