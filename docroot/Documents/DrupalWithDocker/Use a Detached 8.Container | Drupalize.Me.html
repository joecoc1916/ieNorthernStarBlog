Use a Detached Container

Add to queue     Share
Last updated November 11, 2019
Backend and Infrastructure

Running a container interactively can be useful, but often it's not what we really need. A web server stack is made up of several components such as the Linux OS, the Apache web server, a PHP runtime, and a database such as MySQL. Collectively, we call this a LAMP stack. If we were to run these in Docker with what we now know, we'd have to keep open several terminal windows!

Obviously that's not what we want to do. Instead, we want to run the containers in the background. That way, we can use them like we would any web server. Fortunately, Docker makes running and managing a container in the a background easy with just a few commands.

In this tutorial, we'll:

  • Start a container in the background
  • Use docker ps to list running Docker containers
  • Use docker run to enter a container running in the background
  • Use docker kill to stop a container running in the background

Goal

Start, access, and manage containers running in the background.

Prerequisites

  • Run a Container Interactively

Starting a container in background

Before, we used the docker run command to start a container in the background, with the -i and -t switches to run it interactively. To run a container in the background, we only need to specify a different switch:

docker run -d image_name /path/to/command

The -d switch -- 'd' for "detached" -- instructs the docker command to run the container in the background. Instead of running /bin/bash, let's run a different command in the container that will not need our interaction:

$ docker run -d debian sleep infinity
142a93d9ebbc500e131d8ca8767ccc841cfc60f8e00f08d85fbf46286f92ad80

$

When using -d, we're not dropped into an interactive session with the container. Instead, a long string of characters is displayed, and control of the terminal is returned to us.

Container IDs

When running a detached container, we need a way to uniquely refer to it. We can't just call it the container name we passed to docker run. If we have two debian containers running in the background, how can we be sure which instance of debian we're referring to?

The long string of characters returned by docker run -d is the container ID. This is a Universally Unique Identifier (UUID) that not only refers to a particular container instance on your system but any instance anywhere.

List running containers

When we run a detached container, we're given back control of the terminal window. We could even close the terminal window and the container would continue to run in the background until we explicitly stop it. So how can we know what containers are currently running? The docker command does more than just docker run. It provides several subcommands that allow us to interact with Docker.

To see what containers are running, we use the docker ps command:

$ docker ps
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES
142a93d9ebbc        debian              "sleep infinity"    3 seconds ago       Up 2 seconds                            vibrant_noether

The docker ps command tells us several things:

  • The IDs of the containers currently running
  • The image from which those containers were created
  • The command we're running in the containers
  • How long ago the container was created
  • If the container is running
  • Any network ports the container is using
  • The local instance name of the container

You might have noticed that docker ps might remind you of the UNIX ps command for listing processes. This is an intentional design choice on the part of Docker. If you're familiar with other UNIX commands, you'll be able to quickly pick up the docker versions.

Enter into a running container

So we know that our container is running in the background. What if we need to enter into a running container interactively? For that, we use the docker exec command:

docker exec -it container_id /path/to/command

This looks very similar to the docker run command, but instead of passing an image name, we're specifying the container ID. Let's do that with our running container:

$ docker exec -it 142 /bin/bash
root@142a93d9ebbc:/#

The above enters into our running container and executes the /bin/bash command. We specify -i and -t like we did with docker run to make sure we get an interactive session. Notice that we don't need to specify the entire container ID, only the first few characters that uniquely identify it on our system. Usually the first three or four characters are all we need.

The command to run in the container we pass to docker exec doesn't need to be the same as the command we used in docker run. In fact, it often isn't. If we were running an Apache/PHP container, for example, we might use docker exec to run utilities like drush.

We can exit the interactive docker exec session just like we did with docker run, by using the exit command:

$ docker exec -it 142 /bin/bash
root@142a93d9ebbc:/#

...

root@142a93d9ebbc:/# exit

$

Stop a detached container

Stopping a detached container is also simple:

docker kill container_id

The docker kill command stops the container with the given container ID. We can then verify the container is no longer running using docker ps:

$ docker kill 142
142

$ docker ps
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES

$

Notice that there are no items in the list returned by docker ps. This indicates that we have no containers running.

Stop multiple containers

We can stop multiple containers using the docker kill command as well. First, let's start up a few containers:

$ docker run -d debian sleep infinity
166fa164b2b606673d3997b62e1c621f8f7ef8c0a23769cd52a75c83ae9fa5ac

$ docker run -d debian sleep infinity
fa020499ce5a383a6c64cbe0955c460527787eed68826dc4fb16d7983309499b

$ docker ps
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES
fa020499ce5a        debian              "sleep infinity"    6 seconds ago       Up 5 seconds                            loving_hypatia
166fa164b2b6        debian              "sleep infinity"    8 seconds ago       Up 7 seconds                            condescending_kowalevski

Now to stop them, we only need to pass each container ID to docker kill:

$ docker kill fa0 166
fa0
166

$ docker ps
CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES

$

Stop all containers system-wide

What if we need to stop all the containers on our system? Copying and pasting container IDs can be both tedious and error-prone. Fortunately, we can use a bit of terminal wizardry to take care of this for us.

The docker ps command by default prints a human-readable table of containers and statuses. It also has a "quick" mode that can be used by specifying the -q switch:

$ docker ps -q
1909466f4985
fcc548fb2547
a0bcc03f7721

The quick mode only outputs the IDs of running containers. This makes it perfect for scripting. It also makes it easy for us to do a one-line kill of all containers on our system:

$ docker kill `docker ps -q`

This runs both docker ps and docker kill at once. Since the docker ps command is listing all the IDs of running containers on the system, docker kill goes on to stop each container.

Note that the above uses backticks, which may not be correct for your shell or available on your keyboard depending on language. An alternative form is to use a subshell:

$ docker kill $(docker ps -q)

This does the same thing, but might be easier for you to type.

Recap

In this tutorial, we learned how to run containers in the background using the -d switch. We also explored how to use several other docker subcommands such as ps to list running containers, and kill to stop a detached container. Finally, we saw how we could combine both docker kill and docker ps into a single command to stop all containers on the system.
